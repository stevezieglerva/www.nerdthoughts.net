We're building a geographic information system (GIS) to help transportation planners do initial scoping and see high-level impacts on the environment and ecology.

It's a long story, but our application doesn't have a final home or owner yet. Based on that, we're trying to design something with a small footprint, low operating cost, but also fits nicely with some of the industry standard tools used by state departments of transportation and city planning organizations.
Eco-Plan Demo Page

Having some APIs to expose existing government geoprocessing would really help. Let's start with some background.

Open Data Before It Was Cool

The federal GIS communities were publishing important public data  before it was cool or a presidential mandate. Agencies like the Environmental Protection Agency, US Geological Survey, US Fish and Wildlife, Forest Service, National Park Service, and the Census Bureau have been publishing maps and the underlying GIS data for a long time. They are now also serving up the map data via standard APIs to allow people to integrate the data into their websites, apps, or maps.

Hard Part Is Done

There several existing federal web applications that allow you to draw a line or shape of possible project (road, power line, facility) and see the possible environmental and ecological impacts of the project. The newer ones also include some modeling to provide some scale of the impacts. 

EPA's NEPAssist allows a user to draw a project like a road and then see possible environmental impacts with federal lands, impaired streams and waterways, and historical places.

Fish and Wildlife's IPaC tool has similar functionality, but with a focus on ecology and endangered species.

Enegy's EISPC tool gives a neat summary pie chart of how to develop with caution or extreme caution.

These web apps are great resources.  If you're human.

Wrap It In An API

But, if you are another web app, you can't use the geoprocessing. It wouldn't be hard to wrap the existing geoprocessing in a REST API, especially since it's already being served via a web app. Each API could accept the points of a line or polygon via POST parameters and return JSON data of the intersection layers.  How cool would it be to have http://www.epa.gov/compliance/nepa/api/intersect and http://ecos.fws.gov/ipac/api/intersect?
