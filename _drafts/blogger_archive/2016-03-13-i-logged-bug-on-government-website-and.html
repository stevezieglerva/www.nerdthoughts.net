I recently logged a bug for GSA's 18F blog website. I noticed on my RSS reader, Feedly, that the URL in the RSS title was incorrect. It actually duplicated the domain name. It seemed like a simple issue to fix so I decided to try to log it.

Like all 18F work, the website code is hosted in a GitHub repo and has a publicly available Issues page. I poked around at the source code wondering if I could find where the error occurred. I was going to reference the source code and suggest a possible fix to save them time, but the RSS file and properties seemed to be generated externally. So, I just logged the bug (issue 1497) and waited to see what happened.



Within an hour, a developer chimed in suggesting a possible source of the bug. The bug was fixed and the new site deployed to prod the next day. It was a really cool experience and led me to several observations:

  • I expected 18F to respond quickly - Even just a few years ago, I never would have bothered to try to report an issue on a government website. I wouldn't have found an easy way to report the issue and doubt it would have gotten fixed anyways. But, in this case, I expected 18F to respond quickly. They have the processes, development tools, and streamlined website in which they have the ability to respond quickly. I also assumed they would appreciate public comment on their site given their focus on transparency.
  • There are real people behind the website - One cool thing I didn't expect was the personal aspect of the process. Given the use of GitHub, I could see real people discussing the issue and fixing it in real time. I felt very connected to the process, even as a complete outsider.
  • Participating is believing - I've been following 18F since their inception. While I've read numerous posts on their belief in transparency, it didn't really sink in until I participated in the process. While I know this was a simple fix on just a blog site, it's still a huge advancement in the way government treats IT.
