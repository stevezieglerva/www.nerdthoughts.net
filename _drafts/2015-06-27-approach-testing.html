At work, we're in the middle of a coding challenge as part of a government procurement. It's a great exercise in design, UX, and development. The official challenge is to do something with data from https://open.fda.gov. The unofficial challenge is to go from idea generation to working code in 10 days. It raised a running discussion with the team about how much to design before coding. 



These days, I'm mostly a pointy-haired boss. But, on this team, I'm acting as a Deliver Manager and a data person doing some actual scripting. I had a frustrating moment when trying to pull some data that reminded me of the value of coding early. I had planned to use the API to pull some unique data and load it into Elasticsearch. I was confident it would be easy because I'd done similar things many times before. It wasn't until I started that I realized some arbitrary API limits and throttling made it much more difficult than I had planned.

This all occurred with having the developer (me) in all of the discussions and decisions. I estimated two hours to do the work. Four hours into it, I had only accomplished 25% of what I wanted to and was left with doubts about the approach. Luckily, we had designed and coded just enough to find these gotchas and further iterations proved that the approach was solid.

It's got me thinking about how important User Testing is to UX specialists before making too many design decisions with the team. Approach Testing (algorithm, data, tech stack, mobile use) should be just as important to the team before making too many design decisions. It's the same thing as Code First, vertical slices, iterating, Fail Fast. But, Approach Testing feels more relatable when working in a heavy UX environment.
